# 从 0 到 1 做一个可评测的企业知识库客服 Agent

普通 RAG demo 通常只展示“输入问题、返回回答”。这对作品集不够，因为企业客服场景真正关心的是答案是否可追溯、系统是否知道自己不知道、错误问题是否能进入人工流程，以及每一次回答能否被复盘。

EvalSupport AI 的目标不是做一个更会聊天的机器人，而是把 RAG 做成一个可部署、可测试、可观测、可评测的 AI 应用系统。

## 为什么普通 RAG demo 不够

客服系统里的错误回答会直接影响用户决策。只返回一段自然语言有几个问题：

- 面试官无法判断回答来自哪里。
- 系统无法区分“没查到”和“模型编了一个答案”。
- 没有会话记录就无法复盘失败案例。
- 没有固定测试集就无法比较 prompt、模型和检索参数。

所以 MVP 先实现工程闭环：文档上传、索引、检索、带引用回答、兜底工单、会话记录和后台查看。

## 如何设计来源引用

索引阶段把文档切成 chunk，写入 `document_chunks`，每个 chunk 保留：

- 所属文档
- 页码或位置
- 原文片段
- embedding
- metadata

检索阶段把 Top K chunk 映射成 `S1/S2/...`。Prompt 要求模型的事实性结论必须使用 `[S1]` 这样的引用。服务端不会盲信模型输出，而是用正则提取引用标签，并确认标签来自本次检索结果。

如果回答没有有效引用，系统会把它标记为异常并创建工单。

## 如何判断模型“不知道”

MVP 使用三条保守规则：

- Top 1 相似度低于阈值时拒答。
- 没有可引用 chunk 时拒答。
- 模型回答没有有效引用时拒答。

拒答不是失败，而是客服系统的正常分支。系统会返回确定性兜底文案，并在 `tickets` 表创建工单，后台可以看到原问题、触发原因和 AI 初步判断。

## 如何做 AI 应用评测

EvalSupport AI 把评测结果存成 `eval_runs`，Supabase 模式还预留了 `eval_cases`：

- 本地 Demo 使用 20 条固定问题，包含 12 条可回答、5 条应拒答、3 条干扰问题。
- `eval_runs` 保存一次运行的指标和逐条结果。
- `eval_cases` 用于后续 Supabase 自定义评测集。

当前本地评测会统计：

- 引用正确率
- 拒答正确率
- 回答可用率

这些指标能用来比较不同模型、不同阈值、不同 chunk 策略和不同 prompt 版本。后续扩展方向是把固定本地测试集升级成可在后台维护的 Supabase 自定义评测集。

## 如何把大模型能力变成工程系统

这个项目的关键不是“调用一次模型”，而是把模型放进可控流程：

- 上传和索引是可追踪状态机。
- 检索结果是结构化来源。
- Prompt 输出有服务端校验。
- 低置信度有工单兜底。
- 每次回答记录耗时、token、模型和引用。
- 核心逻辑有单元测试。

这样的项目更接近真实 AI 应用工程：不是追求一次炫酷回答，而是让回答可以被解释、被复盘、被评测、被改进。
